title: "POST /api/licenses/offline/deactivation — Offline deactivation"
sidebarTitle: "POST /api/licenses/offline/deactivation"
description: "Submit an offline deactivation request to free an air-gapped machine's slot. Consumes the embedded refresh token; replaying the request returns a stale error."
---

Use this endpoint to deactivate a machine that cannot reach the internet directly. Your plugin generates and exports a deactivation request file on the offline machine; you submit that file from any connected device to free the activation slot. The server extracts the embedded refresh token and `machine_id` from the request, then runs the same logic as the standard deactivation endpoint. Because the refresh token is consumed and the machine token version is incremented, replaying the same request file returns a `stale_refresh_token` error.

**Endpoint:** `POST /api/licenses/offline/deactivation`

**Authentication:** None — authentication credentials are embedded in the offline deactivation request file.

## Request

<ParamField body="request" type="string" required>
  The full text of the offline deactivation request exported from the target machine. This is a signed, encoded payload generated by the plugin SDK. It embeds the machine's `refresh_token`, `machine_id`, and a `request_id`.
</ParamField>

## Response

<ResponseField name="offline_deactivation" type="boolean">
  Always `true` on a successful response.
</ResponseField>

<ResponseField name="request_id" type="string">
  The unique ID extracted from the deactivation request file, confirming which request was processed.
</ResponseField>

<ResponseField name="machine_id" type="string">
  UUID of the machine that was deactivated.
</ResponseField>

<ResponseField name="deactivated" type="boolean">
  `true` if the machine was successfully deactivated.
</ResponseField>

<ResponseField name="machines_remaining" type="number">
  Remaining active machine slots on the license after this deactivation.
</ResponseField>

## Error codes

| Code | Status | Meaning |
|------|--------|---------|
| `invalid_json` | 400 | Request body is not valid JSON. |
| `request is required` | 400 | The `request` field is missing or empty. |
| `invalid_offline_deactivation_request` | 400 | The `request` text could not be decoded or its contents are malformed. The error message may provide a more specific reason. |

Errors from the underlying deactivation call are forwarded directly. These include:

| Code | Status | Meaning |
|------|--------|---------|
| `invalid_token` | 401 | The refresh token embedded in the request file is invalid or expired. |
| `stale_refresh_token` | 401 | The token version in the request file is behind the current machine record — the request file has already been used. |
| `machine_id_mismatch` | 403 | The `machine_id` in the request file does not match the token's claim. |
| `already_deactivated` | 409 | The machine was already deactivated before this request was submitted. |
| `machine_not_found` | 404 | No machine found for the `machine_id` embedded in the request. |

<Warning>
  The deactivation request file contains a live refresh token. Treat it as a sensitive credential — do not share it or expose it in logs. Once the file is successfully submitted, it is permanently invalidated and cannot be replayed.
</Warning>

<Note>
  After a successful offline deactivation, import the confirmation back to the target machine using your plugin SDK so it can clear its local license state. The activation slot is freed immediately upon submission.
</Note>

## Example

**Request**

```json
POST /api/licenses/offline/deactivation
Content-Type: application/json

{
  "request": "eyJyZXF1ZXN0X2lkIjoicmVxXzAxOTIuLi4iLCJtYWNoaW5lX2lkIjoiMDE5MjAw..."
}
```

**Response (200)**

```json
{
  "offline_deactivation": true,
  "request_id": "req_01920000aabbccdd",
  "machine_id": "01920000-aaaa-bbbb-cccc-dddddddddddd",
  "deactivated": true,
  "machines_remaining": 2
}
```

**Response (401 — already used)**

```json
{
  "error": "stale_refresh_token"
}
```
