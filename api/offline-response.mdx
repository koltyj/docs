---
title: "POST /api/licenses/offline/response — Exchange challenge"
sidebarTitle: "POST /api/licenses/offline/response"
description: "Exchange an offline challenge and a license key for a signed license file and refresh token. Challenges are single-use — replaying one returns a 409 error."
---

Use this endpoint to activate a license on an air-gapped or temporarily offline machine. Your plugin exports an offline challenge file on the target machine; you submit that challenge alongside the license key from a connected device to receive an importable activation response. The server runs the same entitlement, machine slot, and signing logic as the standard activation endpoint and records the `challenge_id` to prevent replay attacks.

**Endpoint:** `POST /api/licenses/offline/response`

**Authentication:** None — the license key and challenge file together serve as the credential.

## Request

<ParamField body="challenge" type="string" required>
  The full text of the offline challenge exported from the target machine. This is a signed, base64-encoded payload generated by the plugin SDK on the offline machine. It embeds the machine's `fingerprint`, `components`, `product_id`, `hostname`, `platform`, `challenge_id`, and an `expires_at` timestamp.
</ParamField>

<ParamField body="license_key" type="string" required>
  The plaintext license key to activate against the challenge.
</ParamField>

## Response

<ResponseField name="offline_activation" type="boolean">
  Always `true` on a successful response.
</ResponseField>

<ResponseField name="challenge_id" type="string">
  The unique ID from the decoded challenge, confirming which challenge was consumed.
</ResponseField>

<ResponseField name="expires_at" type="string">
  ISO 8601 timestamp from the challenge indicating when the challenge itself would have expired. The response remains valid for import regardless of this field.
</ResponseField>

<ResponseField name="machine_id" type="string">
  UUID of the activated machine record. Store this on the target machine for use with offline deactivation.
</ResponseField>

<ResponseField name="machines_remaining" type="number">
  Remaining active machine slots on the license after this activation.
</ResponseField>

<ResponseField name="license_file" type="string">
  Ed25519-signed PEM license file to import on the target machine.
</ResponseField>

<ResponseField name="refresh_token" type="string">
  JWT refresh token bound to the activated machine. Import this alongside the license file — it is required for offline deactivation and for validation once the machine is back online.
</ResponseField>

<ResponseField name="license" type="object">
  Summary of the license state at activation time. Contains the same fields as the `/api/licenses/activate` response: `status`, `expiry`, and `entitlements`.
</ResponseField>

## Error codes

| Code | Status | Meaning |
|------|--------|---------|
| `invalid_json` | 400 | Request body is not valid JSON. |
| `challenge and license_key are required` | 400 | One or both required fields are missing or empty. |
| `invalid_offline_challenge` | 400 | The `challenge` text could not be decoded or its signature is invalid. The error message may provide a more specific reason. |
| `product_not_found` | 404 | The product embedded in the challenge does not exist for this tenant. |
| `offline_challenge_replayed` | 409 | A response has already been issued for this `challenge_id`. Generate a new challenge on the target machine. |

Activation errors from the underlying activation call are also forwarded directly. These include `license_not_found`, `license_not_active`, `entitlement_mismatch`, and `machine_limit_reached` — see the [activate endpoint](/api/activate) for their meanings.

<Warning>
  Each challenge is **single-use**. Once a successful response is issued, any subsequent attempt to redeem the same challenge returns `offline_challenge_replayed` (409). If the import fails after the exchange, generate a new challenge on the target machine.
</Warning>

<Note>
  Challenges embed an `expires_at` timestamp. Submit the challenge before that time — expired challenges are rejected during decoding even if they have not been replayed.
</Note>

## Example

**Request**

```json
POST /api/licenses/offline/response
Content-Type: application/json

{
  "challenge": "eyJjaGFsbGVuZ2VfaWQiOiJ1dWlkLWhlcmUiLCJwcm9kdWN0X2lkIjoiMDE5MjAwMDAuLi4ifQ==...",
  "license_key": "SWVL-XXXX-YYYY-ZZZZ"
}
```

**Response (200)**

```json
{
  "offline_activation": true,
  "challenge_id": "chal_01920000aabbccdd",
  "expires_at": "2024-06-02T12:00:00.000Z",
  "machine_id": "01920000-aaaa-bbbb-cccc-dddddddddddd",
  "machines_remaining": 1,
  "license_file": "-----BEGIN LICENSE FILE-----\n...\n-----END LICENSE FILE-----",
  "refresh_token": "eyJhbGciOiJFZERTQSJ9...",
  "license": {
    "status": "active",
    "expiry": null,
    "entitlements": ["my-plugin:full"]
  }
}
```

**Response (409 — replayed)**

```json
{
  "error": "offline_challenge_replayed"
}
```
