The `clients/juce/` directory contains everything your C++ plugin needs to talk to the Swivel Licensing API. It ships as a small set of headers and source files you add directly to your JUCE project — no external package manager required. This page covers the classes the client provides, how to add them to your build, and how to wire them into your `AudioProcessor`.

## What the client provides

<CardGroup cols={2}>
  <Card title="SwivelLicensingClient" icon="network-wired">
    The low-level API client. Accepts an injected `HttpTransport&` and exposes one method per endpoint: `activateOnline`, `validateOnline`, `refreshOnline`, `deactivateOnline`, `exchangeOfflineChallenge`, and `exchangeOfflineDeactivationRequest`. It also provides static helpers for generating offline blobs without a network call.
  </Card>
  <Card title="SwivelLicenseSession" icon="layer-group">
    A higher-level wrapper that combines `SwivelLicensingClient` with a `LicenseStorage&` to manage the full persist/validate/refresh/deactivate lifecycle. Use this in your `AudioProcessor` instead of calling `SwivelLicensingClient` directly.
  </Card>
  <Card title="PlatformLicenseStorage" icon="hard-drive">
    A concrete `LicenseStorage` implementation. It stores machine ID and signed license state in a product-scoped JSON file under the Swivel Audio app-data directory, and stores refresh tokens through macOS Keychain or Windows Credential Manager.
  </Card>
  <Card title="LicenseFilePolicy" icon="shield-check">
    A standalone verifier for signed license files. Call `verifyLicenseFilePolicy` with the PEM, your Ed25519 public key hex, and a `LicenseFilePolicyContext` to check the file locally — no server call needed.
  </Card>
</CardGroup>

## Class signatures

### Core types

```cpp
namespace swivel::licensing
{

struct MachineComponent
{
    juce::String name;
    juce::String value;
};

struct ActivationRequest
{
    juce::String licenseKey;
    juce::String productId;
    juce::String fingerprint;
    juce::Array<MachineComponent> components;
    juce::String hostname;
    juce::String platform;
};

struct LicenseResponse
{
    bool ok = false;
    int statusCode = 0;
    juce::String error;
    juce::String machineId;
    int machinesRemaining = 0;
    juce::String licenseFile;
    juce::String refreshToken;
    juce::var raw;
};

struct LocalLicenseState
{
    juce::String machineId;
    juce::String licenseFile;
    juce::String refreshToken;
    juce::String trustedIssuedAt;
};

} // namespace swivel::licensing
```

### HttpTransport and LicenseStorage interfaces

```cpp
class HttpTransport
{
public:
    virtual ~HttpTransport() = default;
    virtual LicenseResponse postJson (const juce::String& path,
                                      const juce::var& body,
                                      const juce::String& bearerToken = {}) = 0;
    virtual LicenseResponse getJson (const juce::String& path,
                                     const juce::String& bearerToken) = 0;
};

class LicenseStorage
{
public:
    virtual ~LicenseStorage() = default;
    virtual bool read (LocalLicenseState& state) const = 0;
    virtual bool write (const LocalLicenseState& state) = 0;
    virtual void clear() = 0;
};
```

### SwivelLicensingClient

```cpp
class SwivelLicensingClient
{
public:
    explicit SwivelLicensingClient (HttpTransport& transport);

    LicenseResponse activateOnline (const ActivationRequest& request);
    LicenseResponse validateOnline (const juce::String& refreshToken,
                                    const juce::String& productId,
                                    const juce::String& fingerprint,
                                    const juce::Array<MachineComponent>& components);
    LicenseResponse refreshOnline (const juce::String& refreshToken);
    LicenseResponse deactivateOnline (const juce::String& refreshToken,
                                      const juce::String& machineId);
    LicenseResponse exchangeOfflineChallenge (const juce::String& challenge,
                                              const juce::String& licenseKey);
    LicenseResponse exchangeOfflineDeactivationRequest (const juce::String& request);

    static juce::String createOfflineChallenge (
        const ActivationRequest& request,
        juce::RelativeTime ttl = juce::RelativeTime::hours (24));
    static juce::String createOfflineDeactivationRequest (
        const juce::String& machineId,
        const juce::String& refreshToken,
        juce::RelativeTime ttl = juce::RelativeTime::hours (24));
};
```

## Integration steps

<Steps>
  <Step title="Add clients/juce/ to your CMake project">
    Copy the `clients/juce/` source files into your plugin tree (or reference them via `add_subdirectory`). The `CMakeLists.txt` expects either a `SWIVEL_JUCE_DIR` cache variable or a `JUCE_DIR` environment variable pointing at a local JUCE checkout, and it requires `libsodium` via `pkg-config`.

    ```cmake
    cmake_minimum_required(VERSION 3.22)
    project(MyPlugin VERSION 1.0.0 LANGUAGES C CXX)

    # Point at your JUCE checkout
    set(SWIVEL_JUCE_DIR "/path/to/JUCE" CACHE PATH "")

    # Add the Swivel client as a subdirectory
    add_subdirectory(clients/juce SwivelLicensingClient)

    # Pass your Ed25519 public key at configure time
    set(SWIVEL_LICENSING_SIGNING_PUBLIC_KEY "<your-hex-public-key>" CACHE STRING "")

    juce_add_plugin(MyPlugin ...)

    target_link_libraries(MyPlugin PRIVATE
        SwivelLicensingClient
        SwivelLicenseFilePolicy
    )
    ```

    On macOS, the build links `Security.framework` and `CoreFoundation.framework` automatically. On Windows it links `Advapi32` for Credential Manager access.
  </Step>

  <Step title="Choose or implement LicenseStorage">
    For most shipping plugins, use `PlatformLicenseStorage` directly:

    ```cpp
    // Construct once; keep alive for the plugin's lifetime
    swivel::licensing::PlatformLicenseStorage storage { "my-plugin" };
    ```

    Pass your plugin's product slug as the first argument. The storage layer creates a product-scoped JSON state file under the Swivel Audio app-data directory and stores the refresh token separately through the platform secure-token boundary — the token never appears in the JSON file on disk.

    <Note>
      `PlatformLicenseStorage` only supports macOS and Windows. On other platforms, token persistence is compiled out. This is sufficient for harness and CI work but not for a shipping plugin targeting those platforms.
    </Note>

    If you need custom behavior — per-DAW sandboxing, atomic writes, or a different storage path — implement the `LicenseStorage` interface yourself:

    ```cpp
    class MyPluginStorage final : public swivel::licensing::LicenseStorage
    {
    public:
        bool read (swivel::licensing::LocalLicenseState& state) const override { /* ... */ }
        bool write (const swivel::licensing::LocalLicenseState& state) override { /* ... */ }
        void clear() override { /* ... */ }
    };
    ```
  </Step>

  <Step title="Inject transport">
    Use `UrlHttpTransport` for standard HTTPS production traffic:

    ```cpp
    swivel::licensing::UrlHttpTransport transport {
        juce::URL { "https://api.swivel-audio.com" }
    };
    ```

    For local development or staging, pass an `Options` struct:

    ```cpp
    swivel::licensing::UrlHttpTransport::Options opts;
    opts.allowLocalHttp = true; // only for localhost testing

    swivel::licensing::UrlHttpTransport transport {
        juce::URL { "http://localhost:3000" },
        opts
    };
    ```

    To use your own networking layer, subclass `HttpTransport` and pass a reference to `SwivelLicensingClient`.
  </Step>

  <Step title="Wire SwivelLicenseSession into your AudioProcessor">
    Construct `SwivelLicensingClient` and `SwivelLicenseSession` in your `AudioProcessor` subclass. Keep both objects alive for the processor's lifetime:

    ```cpp
    class MyPluginProcessor : public juce::AudioProcessor
    {
    public:
        MyPluginProcessor()
        {
            transport = std::make_unique<swivel::licensing::UrlHttpTransport>(
                juce::URL { "https://api.swivel-audio.com" });
            storage = std::make_unique<swivel::licensing::PlatformLicenseStorage>("my-plugin");
            client  = std::make_unique<swivel::licensing::SwivelLicensingClient>(*transport);
            session = std::make_unique<swivel::licensing::SwivelLicenseSession>(*client, *storage);

            // Validate stored license on startup
            auto result = session->validateStored(
                "product-my-plugin",
                collectFingerprint(),
                collectMachineComponents());

            if (result.ok)
                licenseValid = true;
            else
                promptForLicenseKey(); // show activation UI
        }

    private:
        std::unique_ptr<swivel::licensing::UrlHttpTransport>    transport;
        std::unique_ptr<swivel::licensing::PlatformLicenseStorage> storage;
        std::unique_ptr<swivel::licensing::SwivelLicensingClient>  client;
        std::unique_ptr<swivel::licensing::SwivelLicenseSession>   session;
        bool licenseValid = false;
    };
    ```

    Common error codes returned in `LicenseResponse::error` that require a new activation prompt: `MACHINE_DEACTIVATED`, `INVALID_TOKEN`.
  </Step>

  <Step title="Handle activation in your UI">
    When the user submits a license key, build an `ActivationRequest` and call `activateAndStore`:

    ```cpp
    swivel::licensing::ActivationRequest req;
    req.licenseKey  = licenseKeyFromUser;
    req.productId   = "product-my-plugin";
    req.fingerprint = collectFingerprint();
    req.components  = collectMachineComponents();
    req.hostname    = juce::SystemStats::getComputerName();
    req.platform    = juce::SystemStats::getOperatingSystemName();

    auto result = session->activateAndStore (req);

    if (result.ok)
    {
        licenseValid = true;
        // SwivelLicenseSession has already persisted licenseFile and refreshToken
    }
    else
    {
        showError (result.error);
    }
    ```

    `activateAndStore` calls `activateOnline` and then immediately writes the returned `licenseFile` and `refreshToken` to your `LicenseStorage` — you do not need to write them yourself.
  </Step>

  <Step title="Block audio processing when unlicensed">
    In `processBlock`, zero the output buffer when `licenseValid` is `false`:

    ```cpp
    void processBlock (juce::AudioBuffer<float>& buffer,
                       juce::MidiBuffer& /*midi*/) override
    {
        if (!licenseValid)
        {
            buffer.clear();
            return;
        }
        // ... normal DSP ...
    }
    ```
  </Step>
</Steps>

## Platform secure storage

`PlatformLicenseStorage` separates machine state from the refresh token at the storage layer:

| Platform | State file | Refresh token |
|---|---|---|
| macOS | Product-scoped JSON in Swivel Audio app-data | macOS Keychain (service: `com.swivelaudio.licensing`) |
| Windows | Product-scoped JSON in Swivel Audio app-data | Windows Credential Manager |
| Other | Product-scoped JSON in Swivel Audio app-data | **No token persistence** |

<Warning>
  On unsupported platforms the refresh token is never written to disk. The plugin will require re-activation on every launch. This is intentional for harness/CI builds but is not acceptable for a shipping plugin.
</Warning>

## LicenseFilePolicy checks

`verifyLicenseFilePolicy` applies the following checks in order. A failure in any check produces a `LicenseFilePolicyResult` with `ok = false` and a machine-readable `code`:

1. **Signature validity** — Ed25519 signature over the PEM payload matches the provided `publicKeyHex`.
2. **Tenant binding** — `expectedTenantId` matches the value embedded in the license file.
3. **Product binding** — `expectedProductId` and `expectedProductSlug` match.
4. **Entitlement binding** — `requiredEntitlement` is present in the file's entitlements list.
5. **Machine binding** — `expectedMachineFingerprint` matches the machine binding in the file.
6. **Expiry** — the file's expiry timestamp has not passed (checked against `LicenseFilePolicyContext::now`).
7. **Future-issued rejection** — `issuedAt` is not more than `maxFutureIssuedAtSkewMs` (default 5 minutes) in the future, guarding against clock manipulation.
8. **Rollback rejection** — `issuedAt` is not earlier than `minimumIssuedAt` from local trusted state, preventing replay of older license files.

The policy vectors in `clients/juce/license-policy-test-vectors.json` cover all eight cases deterministically. Load each PEM from the JSON, supply the included public key, apply the listed context, and assert the expected result.

## Offline challenge flow

For machines without internet access, the plugin generates a self-contained challenge blob and the user carries it to a connected machine for exchange.

<Steps>
  <Step title="Generate the challenge in the plugin">
    Call the static helper with the user's license key and the current machine's `ActivationRequest`. Display the returned string to the user — it is safe to copy as plain text:

    ```cpp
    juce::String challenge = swivel::licensing::SwivelLicensingClient::createOfflineChallenge (
        activationRequest,
        juce::RelativeTime::hours (24));

    // Show challenge to user so they can copy it
    showChallengeCopyUI (challenge);
    ```
  </Step>

  <Step title="Exchange on a connected machine">
    The user submits the challenge string to `POST /api/licenses/offline/response` (or via your activation portal). The server returns a signed license file blob.
  </Step>

  <Step title="Import the response into the plugin">
    The user pastes the response back into the plugin. Call `exchangeOfflineChallenge` to validate and store:

    ```cpp
    auto result = client->exchangeOfflineChallenge (challenge, licenseKey);
    if (result.ok)
        storage->write ({ result.machineId, result.licenseFile, result.refreshToken, {} });
    ```
  </Step>
</Steps>

## Verify integration with the harness

Build and run the full integration harness to confirm your setup is correct before shipping:

```sh
npm run build:juce-client
```

The script resolves JUCE from `SWIVEL_JUCE_DIR` or `JUCE_DIR` and builds four targets:

- **`SwivelLicensingClientHarness`** — drives activate, validate, refresh, deactivate, offline challenge creation/exchange, offline deactivation, token rotation, and storage clearing through injected fake transport and storage.
- **`SwivelPlatformStorageProof`** — writes platform state, stores and rotates the refresh token through Keychain/Credential Manager, confirms the token never appears in the JSON file, then clears both.
- **`SwivelLicensingPluginHarnessProof`** — instantiates `SwivelLicensingPluginHarnessProcessor` and checks that locked audio is cleared, activation unlocks processing, validate/refresh use the stored token, and deactivation relocks audio.
- **`SwivelLicensePolicyVectorsProof`** — loads `license-policy-test-vectors.json` and asserts the expected policy result for each vector in C++.

<Tip>
  Run `SwivelLicensePolicyVectorsProof` as part of your CI pipeline. It requires no network access and confirms your local `verifyLicenseFilePolicy` wiring is correct before deployment.
</Tip>
